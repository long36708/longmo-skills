# 正确使用 key 值规范

## 问题描述

规则组的 `key` 值使用不规范会导致用户配置丢失或规则组冲突。

## 错误示例

```typescript
groups: [
  {
    key: 1,
    // ...
  },{
    key: 2,
    // ...
  },{
    key: 3, // 插入到已存在 key 值序列中
    // ...
  },{
    key: 4,
    // ...
  },{
    key: 5,
    // ...
  },{
    key: 6,
    // ...
  },
],
```

## 正确示例

```typescript
groups: [
  {
    key: 1,
    // ...
  },{
    key: 2,
    // ...
  },{
    key: 4,
    // ...
  },{
    key: 5,
    // ...
  },{
    key: 6,
    // ...
  },{
    key: 7, // 新增规则组使用最大 key 值 +1
    // ...
  },
],
```

## 说明

### key 值规范要求

1. **唯一标识**：`key` 是规则组在列表中的唯一标识，也是客户端禁用/启用此规则组的依据
2. **递增排序**：`groups` 中的规则组按照 `key` 值从上到下递增的方式排序
3. **新增规则组**：新增规则组的 `key` 值使用 `之前最大的 key 值 +1`

### 重要原因

GKD 通过 key 值区分用户手动配置的禁用/启用状态。如果新的规则组使用以前的 key 值，会将以前配置中的禁用/启用状态覆盖到新规则组上。

### 处理缺失 key 值

如果发现以前规则组的 key 值不是连续的（有缺失），说明以前有规则组在新版本中被移除了。新增规则组应该继续使用最大的 key 值 +1，而不是填补缺失的 key 值。

## 实际应用

- 规则类别是通过 `name` 的前几个字符判断的，而不是 `key` 值
- 保持 key 值的连续递增可以避免用户配置冲突
- 移除规则组时，其 key 值会被保留，新增规则组不能复用这些 key 值
